# What's built — Apeiron

This page is Apeiron's index of what is implemented, what is partially implemented, and what is still open work. It is the canonical source for the project's current build state.

## What this index contains

Every load-bearing piece of Apeiron's implementation, grouped by area:

- **Engine core** — the precompute/assemble split, the manifest loader, the try/except isolation wrapper, the file-watch reload, the scene loader, the viewer state model.
- **Node types** — the catalog of node-type implementations under `node_types/`.
- **Renderers** — the catalog of renderer implementations under `renderers/`.
- **Bundle output** — the writer that emits color, depth, optional normal, optional ID, and a manifest, matching the painterly module engine's input contract.
- **Text-renderer surface** — the bidirectional LLM-facing interface for observation, action, inspection, and composition.
- **Build tooling** — scripts that run precompute, watch the graph for edits, and trigger incremental rebuilds.
- **Test suite** — tests covering node-type isolation, renderer-swap, topology walks, aggregator dispatch, and bundle output.

An item is "implemented" when it functions reliably and is in regular use. It is "partially implemented" when parts work but the full design is not in place. It is "open work" when the design exists but no implementation has been built yet.

## Engine core

- **Manifest, NodeInstance, View, Channels, EmitContext** — implemented. The core primitive types in `engine/node.py`. Channels is `Dict[str, Any]`, extensible by name; EmitContext gives nodes access to the engine for child lookup and renderer dispatch. The `View` carries optional dream-mode fields (`gravity_mode`, `gravity_up`, `time`) with defaults preserving existing behavior — added by the dream-features skeleton session.
- **Engine.discover()** — implemented. Walks `node_types/` and `renderers/` for `.py` files, imports each, registers by `manifest().name`. Failures during discovery are recorded but don't crash discovery.
- **Engine.spawn()** — implemented. Creates a node instance from a type name and params. Wraps `build()` in try/except — failures mark the instance `dead` with an error message; the engine keeps running.
- **Engine.assemble()** — implemented. Walks the graph from a root id under a viewer, calls each node's `emit()`, composites results. Module isolation via try/except — broken nodes return placeholder channels.
- **Engine.precompute()** — implemented. Walks every spawned node; for any node-type module exposing `precompute_hook(state, engine, node)`, calls the hook and stores the result at `engine.cache[node_id]`. Wrapped in try/except — a broken hook doesn't block other nodes' precompute. New node-types that want build-time work just expose the hook; no engine change needed.
- **Engine `select_children` dispatch** — implemented in `_emit_node()`. A node-type module exposing `select_children(state, view, engine, node) -> List[str]` controls which connections the engine recurses into for this emit. Aggregator uses this to skip the target's full render at far views — runtime work is not done and thrown away when the precomputed impostor is in use.
- **Engine.reload_type()** — implemented. Hot-reloads a node-type module by re-importing its file fresh (deletes from sys.modules, invalidates importlib caches, evicts stale .pyc, re-runs the discover-time loader) and swapping the registration. Robust against the pyc-cache failure mode where reload_type returned True but the new file content wasn't visible.
- **Engine file-watcher** — implemented in `engine/file_watcher.py`. Polling-based (no external dependency); tracks per-file mtime in `node_types/` and `renderers/`; on new files calls `_load_node_type_file`, on modified files calls `reload_type`, on deleted files logs the deletion (v1 does not unregister). Closes the architectural commitment "Claude Code edits trigger incremental rebuild" at the runtime level — new node-type files written to disk become spawnable without engine restart. Tests in `tests/test_file_watcher.py` (4 tests) verify the new/modified/deleted paths plus the seeded-state guard that prevents redundant re-registration on first poll. The proof of feasibility for the workflow-from-within-Apeiron plan.
- **Engine.sim_precompute()** — implemented as a companion to `precompute()`. Walks every spawned node; for any module exposing `sim_precompute_hook(state, engine, node)`, calls it and stores the result at `cache[node_id + "__sim__"]`. SimulationProbe uses it. Wrapped in try/except so a broken hook doesn't block other nodes. New node-types pre-simulating interactions just expose `sim_precompute_hook` — same pattern as `precompute_hook`. Partially implemented — the surface is wired and tested; mature step()-implementing node-types are future work.
- **Engine.invert_edit(node_id, edit)** — implemented. Walks from `node_id` to the nearest node exposing `invert_hook`; calls the hook; applies the returned parameter delta to the connected Seed; re-triggers the node's precompute_hook. Implementation in `engine/inverse.py` (imported lazily to avoid circular import). Generator demonstrates the surface; future Generator subtypes implement richer `invert_hook` algorithms.
- **Input system (`engine/input.py`)** — implemented as types only. `InputEvent`, `Binding`, `Bindings` (with `Bindings.default()` Minecraft-style profile), `BindingContext`, `ViewMutation`, `apply_mutation()`. No input source is wired (the realtime renderer is future work); the types are usable for headless input simulation and by the future interactive renderer.
- **Scene loader** — implemented as `Engine.load_scene()`. Reads `scenes/<name>.json` and spawns each node listed.
- **Default compositor** — implemented as `Engine._composite_children()`. Z-buffer for color/depth applies the same mask to normal/ids so all geometry channels stay consistent at each pixel (the depth-winner's normal and ids appear, not whoever happened to be composited last). List-concat for text. Future renderer-nodes override this for their sub-graph.
- **Topology transforms** — implemented as `Engine._apply_transform()`. Connections can carry an optional 4x4 transform applied when traversing; identity transforms are the no-op case. Non-identity transforms enable impossible-geometry node-types in future work.

## Node types

Implementations live in [node_types/](node_types/).

- **Cube** — implemented in `node_types/cube.py`. Axis-aligned cube with ray-cast emit() against AABB; produces `color`, `depth`, and `ids` channels. Lambertian-ish shading via dominant-axis approximation. Also exposes `describe()` for the text-renderer.
- **Group** — implemented in `node_types/group.py`. Container that composites its children using the engine's default compositor. No params; connections name children. Demonstrates composition-only node-types.
- **Portal** — implemented in `node_types/portal.py`. Rectangular doorway in the XY plane at z=0; ray-casts against the doorway and uses the "through" child's color for inside-doorway pixels. The 4x4 connection transform on "through" is applied automatically by the engine — the Portal node-type itself adds only rectangle masking on top of existing engine machinery. Demonstrates the topology-over-coordinates architectural commitment: impossible geometries are a node-type, not an engine feature. The `scenes/portal_demo.json` scene plus the `test_portal_renders_through_to_blue_cube` test verify both the parent-frame and through-portal rendering paths.
- **Aggregator** — implemented in `node_types/aggregator.py`. Observes its "target" sub-graph; `precompute_hook` walks the target collecting cube positions and colors, computes centroid + bounding-box + average-color (the BehaviorSummary), caches in `engine.cache`. At emit time, `select_children` returns `[]` when viewer-distance-to-centroid exceeds the threshold (skipping the target's runtime render entirely) and emit returns the colored AABB impostor; otherwise `select_children` returns `["target"]` and emit returns the target's full nine-cube composite. Demonstrates three architectural commitments simultaneously — aggregator-as-node, emergence-at-scale, and precomputation-moves-heavy-work-to-build-time. The `scenes/aggregator_demo.json` scene (3x3 grid of distinctly colored cubes wrapped in an Aggregator) plus four tests (`test_aggregator_precompute_populates_cache`, `test_aggregator_far_view_renders_impostor`, `test_aggregator_near_view_renders_individuals`, `test_aggregator_skips_target_render_at_far_view`) verify the precompute pipeline, the dispatch at both view ranges, and the runtime-work-skipping invariant.
- **Computer** — implemented in `node_types/computer.py`. Owns both a rectangular screen surface in outer world space (XY plane at z=0 in its local frame) AND the camera that renders its "running" sub-graph internally. `select_children` returns `[]` so the engine doesn't redundantly render the sub-graph under the outer view; emit() constructs an internal View from state (position, target, FOV, resolution sized to screen aspect), calls `ctx.engine.assemble(running, internal_view)` to render the sub-graph, then ray-casts the outer view against the screen rectangle and pastes the internal render via UV sampling onto inside-screen pixels. Outside pixels stay transparent so other outer-world geometry composites through naturally. Demonstrates renderer-as-node + screen-region-ownership + recursive worlds (unbounded — a Computer's sub-graph can contain another Computer; the recursion just falls out of the engine's normal traversal). The `scenes/computer_demo.json` scene (red cube in the outer world plus a Computer showing a blue cube on its screen) plus three tests (`test_computer_screen_shows_inner_cube`, `test_computer_skips_default_recursion`, `test_computer_recursion_two_levels`) verify the dispatch, the runtime-work-skipping invariant, and the unbounded-recursion claim with two nested computers.
- **Sphere** — implemented in `node_types/sphere.py`. Ray-cast against a sphere at the local origin; smooth shading from the surface normal's dot product with the camera-forward direction. Same channel contract as Cube. Demonstrates that geometric primitives are independent files; new shapes drop into `node_types/` without engine changes.
- **Plane** — implemented in `node_types/plane.py`. Bounded floor plane in local XZ at y=0; normal +Y; same channel contract. Walls and ceilings via rotation transforms on the connection. Completes the basic primitive set needed for grounded scenes (floor + objects).
- **Light** — implemented in `node_types/light.py`. Directional light source; no visual emit (renders empty channels). At precompute time, registers `{direction, color, intensity}` in the engine's shared `cache["__lights__"]` list. Other renderer-nodes (LambertianShader for now; future point/spot/area lights, shadow casters, etc.) consume from that cache. Demonstrates the "logic-as-node + cache as shared state" pattern: any node-type can register information at precompute that other node-types read at emit.
- **ChatInterface** — implemented in `node_types/chat_interface.py`. Owns a screen rectangle in the outer world that displays the contents of a chat log file. The "side channel" to Claude Code is the file itself: external LLM sessions read it to know what was said and write to it to add new turns; the node visualizes the current contents on each emit. Uses PIL ImageDraw for text-to-pixels with a system-font fallback chain. select_children returns [] (no children to render). Demonstrates the artist-authoring-loop pattern with the authoring tool living as a node inside the system being authored — closes the self-referential loop without engine-level Claude API integration. `scenes/chat_demo.json` plus `logs/sample_chat.txt` show the rendered conversation; `test_chat_interface_renders_log_text` verifies both background and text-stroke pixels are produced.
- **DimensionN** — implemented in `node_types/dimension_n.py`. N-dimensional shape primitive emitting raw N-D vertices and edges on named channels (`verts_nd`, `edges`, `nd_color`). v1 shapes: `hypercube`, `simplex`, `cross_polytope`. Does not itself rasterize — a downstream projector renderer (ProjectorN) consumes the channels. Demonstrates the architecture's "any geometry shape is a node-type file" commitment generalized beyond 3D. Tests verify the 4D hypercube produces 16 vertices and 32 edges, simplex produces N+1 vertices and binom(N+1, 2) edges.
- **Seed** — implemented in `node_types/seed.py`. Parameter holder for procedural generation. State is `{rng_seed, params}`; emit returns empty image channels (data-only node). Consumed by a Generator via the `seed` connection. Future Seed subtypes carry richer parameter shapes (terrain seeds, character seeds, time-evolving seeds); the connection convention stays the same.
- **Generator** — implemented in `node_types/generator.py`. Reads a connected Seed via `precompute_hook`, computes a content spec (v1: list of cube positions, colors, sizes), and inline-rasterizes via emit. **Exposes `invert_hook(state, edit, engine, node)`** — the engine's `invert_edit()` calls this when the viewer mutates generator output; v1's hook handles `{"target": "cube_position", "index": int, "new_value": [x,y,z]}` by updating the seed's `cube_positions` parameter and re-precomputing. Future Generator subtypes implement richer inversion (gradient over parameters, learned models, multi-parameter tuning). The `scenes/seed_world_demo.json` scene plus tests `test_generator_precomputes_specs_from_seed`, `test_generator_emit_renders_cubes`, and `test_invert_edit_updates_seed_and_re_precomputes` verify the precompute, render, and invert paths end-to-end.
- **GravityField** — implemented in `node_types/gravity_field.py`. Region carrying its own gravity vector. `precompute_hook` registers the region+vector in `engine.cache["__gravity_fields__"]`; the helper `active_field(engine, position)` returns the field at a query point. The future interactive renderer's input handler consults this when computing the viewer's effective `gravity_up`. No visual emit. Tests verify registration and lookup.
- **KeyBindings** — implemented in `node_types/key_bindings.py`. Settings-shaped node wrapping an `engine.input.Bindings` table. v1 profile: `"minecraft"` (default). `precompute_hook` registers the node as the active bindings (`cache["__active_key_bindings__"]`). Mouse and controller `look_sensitivity` is a single shared field per the user's "same scale" intent. Test verifies the default profile loads with at least four WASD entries.
- **SimulationProbe** — implemented in `node_types/simulation_probe.py`. Pre-simulates `step()` interactions over a `horizon` at `dt`. `sim_precompute_hook` walks observed nodes, calling their `step()` per tick, building a trajectory stored in `cache[node_id + "__sim__"]`. Future mature node-types implementing `step()` populate richer trajectories. v1's emit exposes the trajectory on a `simulation` channel for future trajectory-rendering renderers (predicted paths, equilibrium markers). Test verifies sim_precompute walks even for targets without step().
- **ChatInterpreter** — implemented in `node_types/chat_interpreter.py`. Parses chat-log messages into engine commands. `precompute_hook` reads new lines from the chat log, classifies each as `"matched"` / `"novel"` / `"noise"`, and (when `claude_connected: true`) writes `requested:` lines back for Claude Code to pick up. v1 ships substring matching against `DEFAULT_COMMANDS`; future parse-pipeline layers are new node-type files (`interpreter_token`, `interpreter_phrase`, etc.) chained via connections. Tests verify classification and the Claude-Code-connected request path.
- **FileSource** — implemented in `node_types/file_source.py`. Reads a file, applies a named parser from `node_types/parsers/`, exposes normalized items on the `items` channel. `precompute_hook` does the read+parse once at build time; `emit()` reads from `engine.cache[node_id]`, keeping per-frame cost constant. Pairs with any items-consuming renderer (ListRenderer is the first). Half of the load-bearing DataSource + Renderer orthogonal primitive that grants Tier A (#001-#005, #009 + #014). Graceful degrade on missing file or unknown parser — items=[] with an error message that downstream renderers display in-place.
- **MCPSource** — implemented in `node_types/mcp_source.py`. Calls an Alethea-cc MCP tool via direct Python import (FastMCP's `@mcp.tool()` decorators are transparent so the underlying functions remain callable as plain Python), optionally applies a parser to the stringified result, exposes items on the `items` channel and the raw tool result on a `raw` channel. The companion to FileSource — together they form the DataSource family. `precompute_hook` runs the actual MCP call so per-frame cost stays sub-ms. Graceful degrade if `mcp` package missing or Alethea-cc not on path: items=[], error message, no crash. Grants wish #001 (MCP adapter) and is the primitive future Email / Calendar / Corpus-browser panels build on.
- **ListRenderer** — implemented in `node_types/list_renderer.py`. Renderer-node that owns a screen rectangle in the outer world; consumes the `items` channel from a connected DataSource via `engine.cache[source_node_id]`; draws a vertical list with status glyphs and titles via PIL ImageDraw. Status-to-glyph and status-to-color maps are exposed params so each panel instance customizes per its source's status taxonomy (Tasks use `[ ]/[x]/[~]`, Wishes use `pending/granted`, Ideas use `pending/resolved`). Reuses the ray-cast + UV-sample screen-rectangle paste primitive shared with ChatInterface and Computer. `describe()` produces text-API output: title + item-count + bulleted item list with glyphs + stable IDs.
- **WorkflowView** — implemented in `node_types/workflow_view.py`. Composite node-type for the maintainer's three-panel workflow surface. Connections via `panel_a`/`panel_b`/`panel_c` (extensible to `panel_d`/`panel_e`), `top_bar`, `chat_bar`, and `full_render`. Mode field (`"panels"` | `"full_render"`) toggles which children render — `select_children` returns panels+bars in panels-mode and only the full-render child in full-render-mode. No own `emit()`: engine's default z-buffer compositor stacks the children. Connection transforms in the scene JSON position the panels side-by-side. Grants wish #005; sets up the future `full_render` toggle (wish #010) at the architectural level. `set_mode(node, new_mode)` is the text-API mutation surface.
- **Parsers** — implemented in `node_types/parsers/`. Each parser exposes one function `parse(text) -> list[dict]` returning normalized items with the shared shape `{id, title, body, status, meta}`. v1 parsers: `tasks` (markdown checkbox checklist with `[ ]/[x]/[~]/[-]`), `wishes` (Apeiron wishlist format with tier-headers tracked as `meta.tier`), `ideas` (Alethea ideas-queue format with Queue/Resolved sections). Auto-discoverable by short name via `node_types.parsers.get_parser(name)`. New panels with novel data shapes write a new parser; old panels are untouched.

## Renderers

Implementations live in [renderers/](renderers/).

- **TextRenderer** — implemented in `renderers/text.py`. The first-class bidirectional LLM interface. Walks its wrapped sub-graph, calls each child's `describe()` (or falls back to type+params), produces structured text output via the `text` channel. Exposes `command_grammar()` listing the text commands the tools layer dispatches.
- **AsciiDebug** — implemented in `renderers/ascii_debug.py`. Renders the depth channel as ASCII art for text-mode topology debugging. Useful for verifying scene geometry from the viewer's position without opening a rendered PNG.
- **PainterlyPostProcessor** — implemented in `renderers/painterly.py`. The first bundle-consuming renderer: wraps a "source" sub-graph, reads its color + ids channels, applies palette reduction (color quantization to N levels per channel) and ID-edge darkening (pixels where the IDs channel changes get multiplied by an edge_darkening factor). Validates the bundle format as a pipeline seam — the same channel shape Apeiron writes to disk also flows through internal renderers. v1 has two effects baked in; v2 can factor each into its own renderer-node and chain them via composition. The `scenes/painterly_demo.json` scene plus `test_painterly_post_quantizes_color_and_marks_edges` verify both effects fire.
- **LambertianShader** — implemented in `renderers/lambertian.py`. Deferred-shading-style renderer-node: wraps a "source" sub-graph, reads its color + normal channels, reads the engine's `cache["__lights__"]` populated at precompute time by Light nodes, and outputs `base_color * (ambient + sum(light_color * intensity * max(0, n · -light_dir)))`. Adding new light types (point, spot, area) is a new node-type registering richer metadata in the same cache — the shader picks them up. The `scenes/lighting_demo.json` scene plus `test_lighting_demo_produces_lit_output` verify the full pipeline (Light registration, source channels consumed, lit output produced).
- **Software raster** — implemented inline as the engine's default compositor (Z-buffer over color/depth). Cubes and other geometric node-types ray-cast in their own `emit()` and the compositor stacks the results. A future explicit `SoftwareRaster` renderer-node can override this for sub-graphs that need different compositing.
- **ProjectorN** — implemented in `renderers/projector_n.py`. Renderer-node that wraps an N-D sub-graph (typically a DimensionN), reads its `verts_nd` and `edges` channels, projects N-D vertices into 3D, and rasterizes edges into the standard color/depth/ids channels via a Bresenham-style line draw. v1 projections: `"drop"` (orthogonal — drop trailing dims) and `"perspective_4d"` (perspective from a 4D viewpoint, the classic "inside-out tesseract" effect). The `scenes/dim4_cube_demo.json` scene plus `test_dim4_cube_scene_assembles` verify the full pipeline; the output is a visible projected tesseract.

## Bundle output

- **`write_bundle()`** — implemented in `engine/bundle.py`. Writes `color.png` (RGB or RGBA), `depth.png` (8-bit visualization), `depth.npy` (raw float depth), optional `normal.png`, optional `ids.png` (16-bit), and `manifest.json` (channel index, view metadata, scene metadata). Text channels land in the manifest directly. Unknown channels save as `.npy` plus a manifest reference, so new channel types pass through forward-compatibly.

## Text-renderer surface

- **Observation channel** — implemented via `TextRenderer.emit()` and `tools/text_test.describe_view()`. Produces a structured text description of the scene at the viewer's position.
- **Action channel** — implemented via `tools/text_test.dispatch_command()` plus the command grammar declared in `renderers/text.py`. Commands: describe, describe-subtree, list-types, list-nodes, spawn, connect, move, look-at, render, render-text.
- **Inspection channel** — implemented via `tools/text_test.describe_scene()` plus list-types and list-nodes commands. Surfaces node state, connection topology, and registered types as text.
- **Composition support** — implemented via the spawn and connect commands plus hot-reload (`Engine.reload_type()`). An LLM can write a new node-type file, trigger a discover/reload, spawn instances, and connect them — all through the text surface.
- **Bundle summary** — implemented as `tools/text_test.summarize_bundle()`. Reads a written bundle directory and produces a text summary of channel statistics for regression testing.

## Build tooling

- **`tools/render.py`** — implemented. CLI entry point that loads a scene, runs precompute/assemble, and writes a bundle to `output/`. Invocation: `python -m tools.render <scene>`.
- **`tools/text_test.py`** — implemented. CLI subcommands plus library functions for text-based testing: `describe`, `view`, `summary`, `command`. Designed so an LLM can verify scene state without opening a rendered image.

- **`tests/test_engine.py`** — implemented with 25 tests covering discovery, spawn, assemble, bundle output, the text-renderer, the command dispatcher, the visibility assertion, module isolation, Portal, Aggregator, Computer, and the PainterlyPostProcessor's quantization + edge-detection effects. Run with `pytest tests/`.
- **`tests/test_dream_features.py`** — implemented with 26 tests covering the dream-mode skeleton: View extension fields, default bindings + apply_mutation, the double-tap-space gravity toggle, DimensionN shape constructors, ProjectorN's 4D-cube end-to-end render, Seed + Generator + invert_edit, GravityField registration and lookup, KeyBindings profile loading, sim_precompute_hook, ChatInterpreter classification + Claude-Code-connected request path, and the three new demo scenes loading cleanly.
- **`tests/test_file_watcher.py`** — implemented with 4 tests covering new/modified/deleted file paths plus the seeded-state guard. The proof-of-feasibility for the workflow-from-within-Apeiron plan.
- **`tests/test_workflow_view.py`** — implemented with 21 tests covering Tier A: parsers (tasks/wishes/ideas with all status glyphs and stable IDs), FileSource (cache integration, missing-file and unknown-parser graceful degrade), MCPSource (unresolvable-tool graceful degrade plus parser-fed-by-tool-result via monkeypatch on the engine's module reference), ListRenderer (items consumed via engine.cache, error rendered inline, describe surfaces glyphs), WorkflowView (select_children for panels-mode and full-render-mode, describe surfaces children inventory, set_mode validation), and end-to-end (workflow_view.json scene loads and renders, panel isolation under sabotaged-source confirmed). All 84 tests (31 original + 26 dream-features + 4 file-watcher + 21 workflow-view + 2 misc) pass with no regressions.

## Realtime renderer

The interactive event-loop driver and backend layer that turn any scene into a live window: WASD/mouse moves the camera, Escape toggles the WorkflowView mode (or quits non-WorkflowView scenes), each frame calls `engine.assemble()` and blits the resulting color channel.

- **`engine/realtime.py`** — implemented. Backend-agnostic `RealtimeDriver` + `WindowBackend` protocol. The driver owns the loop: poll events → resolve via `Bindings` (with `BindingContext` tracking held-keys + last-press-time for double-tap detection) → apply `ViewMutation` to the `View` → call `engine.assemble(root_id, view)` → blit color. Global handlers run before bindings and can consume events (`escape_toggles_workflow_mode_or_quits` is the default). Survives assemble errors by re-blitting the last good frame. Pure logic — testable with a stub backend that records `blit_color` calls and feeds canned `InputEvent`s.
- **`engine/realtime_tk.py`** — implemented. Pure-stdlib backend using `tkinter` + Pillow `ImageTk` for blitting. Translates Tk's callback-driven event model into the driver's poll model via a deque; `normalize_tk_key` maps `"Escape"`/`"Return"`/`"space"`/letter keys onto the canonical names the default `Bindings.default()` table expects. Mouse motion deltas, mouse buttons, and scroll wheel (Windows + X11 conventions) all translate to `InputEvent`. No external dependency — tkinter ships with CPython; Pillow is already required.
- **`tools/realtime.py`** — implemented. CLI entry `python -m tools.realtime <scene> [--width 800] [--height 600] [--backend tk] [--max-frames N] [--fps 60]`. Loads the scene, builds the view from scene metadata + defaults, opens the backend, runs the loop. `--max-frames 1 --fps 0` is the headless smoke test path. Installed as the `apeiron-realtime` console script via pyproject.
- **Backend discovery** — implemented. `available_backends()` returns the names of installed backends best-first (currently `["tk"]`). `make_backend(name=None)` constructs by name or picks the first available. A future pygame backend (better for dream-mode first-person mouse-look with pointer-lock) plugs in by appending `"pygame"` to `available_backends` and a `from engine.realtime_pygame import PygameBackend` branch in `make_backend`.
- **Workflow shell `/realtime` command** — implemented. `tools/workflow/shell.py::cmd_realtime` opens the current (or named) scene in an interactive window; blocks the shell until the window closes. Default scene is `scenes/workflow_view.json`; the shell tracks `last_scene_root`/`last_scene_path` and reuses them. The engine is shared with the shell, so file-watcher hot-reloads from the shell appear in the realtime window's next frame.
- **Escape-toggles-WorkflowView-mode** — implemented. The default global handler toggles WorkflowView's `mode` field between `"panels"` and `"full_render"`. Other scenes treat Escape as quit. This satisfies wishlist #010 (Full-render mode toggle) at the input-handling layer; the full-render scene-tree that the mode switches to remains the connection slot WorkflowView's `full_render` connection points at.
- **Tests** — `tests/test_realtime.py` (19 tests) cover: Tk key-name translation; WASD → ViewMutation through `Bindings.default()`; held-keys + last-press tracking; assemble + blit produces HxWx3 uint8; the engine.assemble survives induced exceptions (driver re-blits last good frame); Escape toggles WorkflowView mode in panels/full_render directions; Escape quits when root is not WorkflowView; the run-loop honors `should_quit`, `max_frames`, and `request_quit`; color normalization (float→uint8, RGBA→RGB, shape validation); backend discovery returns `"tk"` when tkinter is importable; unknown backend names raise. All 19 pass headlessly — no Tk window opens in CI.

The realtime renderer closes the design's "input → view-mutation → render" loop and turns Apeiron from a one-shot bundle renderer into an interactive surface. WorkflowView in panels mode renders the three-panel workflow surface live; the Escape toggle hooks the full-render dream-mode path that future Tier D work fills in (wishlist #024-#030).

## Workflow shell

The interactive Python shell that hosts the workflow-from-within-Apeiron loop. Lives at [tools/workflow/](tools/workflow/). Entry point: `python -m tools.workflow`.

- **SessionManager** — implemented in `tools/workflow/session_manager.py`. Spawns / sends / resumes / archives `claude` CLI subprocesses in stream-json mode (same flags as the cockpit's TypeScript SessionManager: `--print --output-format stream-json --input-format stream-json --include-partial-messages --verbose --permission-mode auto --session-id <uuid>`). Multiplexes subprocess stdout into three channels — `communication` (assistant text), `activity` (tool calls + lifecycle + unknown event types), `diagnostic` (raw stream-json appended to `state/raw_logs/<id>.jsonl`). Per-session silence watchdog emits `silent_too_long` at 5 min. Robust against stream-json schema drift: unknown event types surface as `activity` events with the unknown-type recorded. Tested against a fake-claude script (`tests/fixtures/fake_claude.py`) so test runs don't depend on the real `claude` CLI being installed.
- **Inbox** — implemented in `tools/workflow/inbox.py`. File-based message queue wire-compatible with the Alethea-cc convention at `Alethea-cc/nodes/inbox_msg_*.md` (YAML frontmatter `to: / from: / kind: / summary: / connects_to: / replies_to:`). Auto-detects a sibling Alethea-cc checkout and writes messages into the shared `nodes/` directory by default; falls back to a local `state/inbox/` when no shared checkout is on disk. This is the SAME transport sessions use to message their own subagents — the workflow shell and the spawned sessions share one inbox, with the difference between session-to-subagent and shell-to-session being only the processing step (a session reads via its MCP-wrapped inbox tool; the shell reads via `Inbox.list_all()`).
- **Shell** — implemented in `tools/workflow/shell.py`. Interactive REPL. Boots the Apeiron engine + file-watcher; accepts user input on a background-thread stdin reader so session/inbox/file-watcher events can print to stdout asynchronously. Slash commands: `/help`, `/list [sessions|types|nodes|inbox]`, `/spawn <type> [name] [-- seed]`, `/target <session>`, `/send <session> <msg>`, `/wish <description>`, `/inbox [unread|to <addr>|post ...]`, `/render`, `/dispatch`, `/types`, `/nodes`, `/reload <type>`, `/archive <session>`, `/quit`. Bare text without a leading `/` is sent to the currently-active session (one is auto-spawned by `/wish` when none exists).
- **Wish flow** — `/wish <description>` packs the description into a structured prompt (template lives in `tools/workflow/shell.py::_build_wish_prompt`) that tells the receiving session "you are running inside the Apeiron workflow shell at <repo path>; the file-watcher will hot-reload the new type — no restart is needed." The session writes the new file; the file-watcher detects it and the engine registers it; the next chat message can use the new capability. The end-to-end demo lives as `tests/test_workflow_end_to_end.py::test_fwatch_picks_up_session_written_node_type` and runs with no real `claude` dependency — it drives the SessionManager against the fake-claude fixture, which writes a node-type when given a `WRITE_NODE_TYPE <abs-path>` directive. The test verifies that mid-conversation file writes appear in `engine.types` automatically.
- **State directory** — `state/` (gitignored). Per-session JSONs at `state/workflow/sessions/<uuid>.json`, raw stream-json logs at `state/workflow/raw_logs/<uuid>.jsonl`, archived sessions at `state/workflow/archive/`, local inbox at `state/workflow/inbox/`. Resume after restart picks up the persisted sessions; the workflow shell hydrates them as `idle` until `/send` reactivates them via `--resume`.

The realtime renderer (wishlist #023) and the in-Apeiron 3D visualization of the same surface are still future work. The current shell is the **text-API rendering** of the workflow surface — by the same orthogonality the architecture promises, a future realtime renderer subscribes to the same engine and serves the same scene. The shell delivers the maintainer's "all future work from inside the software" milestone today; the 3D version is a renderer swap when it lands.

## How to modify this page

This page is an [evolving index](https://github.com/liamhodgsonhollingsworth/The-Resonance-Wavefront/blob/main/conventions/evolving_indexes.md).

- **Linking.** When a piece of implementation lands in the repo, add it under the appropriate area with a one-sentence description of what it does and its current state (implemented / partially implemented / open work).
- **Promoting.** A sub-area becomes its own page when it grows enough to warrant separate structure — for example, "Renderers" could split into a top-level Atlas entry once it has many renderer variants.

The trigger for an entry: a piece of work has changed state (implementation has started, is partial, has completed) or has been added to the project. State changes are tracked by appending a dated note rather than overwriting; the history of build state is itself part of the project's record.
