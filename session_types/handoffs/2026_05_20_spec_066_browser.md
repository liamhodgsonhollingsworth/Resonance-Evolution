## What got built

- **`node_types/browser_renderer.py`** — new `BrowserRenderer` node-type.
  Owns a screen rectangle; accepts `url`, `html_string` (override),
  `screen_width/height`, `viewport_width/height`, `refresh_seconds`,
  `background_color`, `backend`. Mirrors the ChatInterface / Computer
  shape; `precompute_hook` caches source mode + optional Playwright
  bitmap; `emit` paints the bitmap (3D path) or a solid screen rect
  (no-bitmap fallback). 2D path is the GUI shell's `web` view kind.
- **`tools/browser.py`** — small primitive: `is_available()`,
  `open_url(parent, url)`, `open_html(parent, html)`,
  `current_url(frame)`, plus `BrowserUnavailableError` with the
  install command in its message.
- **`tools/workflow_gui/view_registry.py`** — added `web` to
  `VALID_KINDS`, extended `ViewSpec` with `url`/`html_string` fields,
  added `web` branch to `as_tabs`, and registered a `Browser` view
  in `default_view_registry()` (kind=`web`, default `about:blank`).
- **`tools/workflow_gui/gui_shell.py`** — `_render_active_tab` now
  routes `web` kind through new `_activate_web(spec)` which packs a
  URL `Entry` + `Go`/`Refresh` buttons + an `HtmlFrame` into the

- **`tools/text_test.py`** — three new SPEC-081 verbs:

## Trust gate composition (SPEC-054)

The new node-type file lives at `node_types/browser_renderer.py` —
inside the default render-trust patterns `node_types/*.py`. So a
paste of a `BrowserRenderer` snippet flows through the engine's
`discover()` → `trust_set.is_trusted(source_id)` exactly like any
other node-type, with the same paste-trust-gate (SPEC-073 follow-up)
catching pastes of unknown type-names. No bypass.

  follow-ups per the design doc.

  sites, the maintainer can add `[extras]` or opt into the
  `webview2` backend via the node's `backend` param (param is wired;
  the backend dispatch is a follow-up).
